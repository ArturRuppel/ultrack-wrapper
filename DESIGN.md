# Ultrack Wrapper — Napari Plugin Design

## Goal

A self-contained napari plugin for interactive 3D nucleus segmentation and
tracking of spinning-disk confocal timelapse data. Uses Cellpose for
segmentation and Ultrack for tracking. The `cellpose_segmentation` project
serves as a reference implementation only — all code is written fresh.

---

## Computational constraints

| Stage | Compute | Napari integration |
|-------|---------|-------------------|
| s00 — Raw data import | Fast (I/O only) | In-process |
| s01a — Cellpose 3D nucleus | **Heavy GPU, minutes–hours** | External only (terminal or generated script) |
| s01b — Cellpose 2D cell | Moderate GPU | External or in-process for small data |
| s02 — Foreground detection | Fast CPU | In-process |
| s03 — Ultrack tracking | Moderate–heavy CPU (ILP) | In-process (threaded) |
| s04 — 2D projection | Fast CPU | In-process |
| s05 — Cell label expansion | Moderate CPU | In-process (threaded) |

**s01a must never run inside napari.** It either opens a new terminal or
generates a script file pointing to input/output directories.

```
      viewers/

```

## Stage design (v1 — start here)

### s02 — Foreground detection (FIRST TO IMPLEMENT)

Goal: determine a good binary foreground mask from Cellpose 3D outputs
(`_dp.tif` and `_prob.tif`). This replaces the old multi-hypothesis
`labels_to_contours` approach with a simpler, more controllable method.

Based on `test_foreground.py` from the reference project, the best approach
uses the **flow magnitude max-Z projection** with thresholding:

```
dP = load(t000_dp.tif)           # (3, Z, Y, X) float32
mag = norm(dP, axis=0)           # (Z, Y, X)
mag_maxz = mag.max(axis=0)       # (Y, X)
foreground = mag_maxz > threshold
```

Thresholding options to expose in the UI:
- **Fixed threshold** on flow magnitude (default: 1.0)
- **Otsu** (adaptive per-frame)
- **Triangle** (adaptive per-frame)

The widget should let the user:
1. Point to a directory containing `t*_dp.tif` + `t*_prob.tif` files
2. Choose thresholding method + parameters
3. Click "Preview" to see foreground overlaid on the max-Z flow magnitude
4. Click "Run all" to process all timepoints and write output

Output: `foreground.tif` — (T, Y, X) uint8 binary mask

This stage runs entirely in-process (fast, CPU only).

## External execution for s01a

### Option A: Launch in terminal

The plugin constructs a command line and opens a new OS terminal:

```bash
python -m ultrack_wrapper.stages.s01a_cellpose_nucleus \
  --input-dir /path/to/raw/ \
  --output-dir /path/to/cellpose_output/ \
  --model nuclei --diameter 17 --anisotropy 1.0 --min-size 500 \
  --timepoints 0 1 2 3 4
```

Platform-specific terminal launch:
- Linux: `gnome-terminal -- bash -c "CMD; read"`
- macOS: `osascript -e 'tell app "Terminal" to do script "CMD"'`
- Windows: `start cmd /k CMD`

After launch, the plugin polls the output directory for new `t*_dp.tif` files
using a `QTimer` (every 5 seconds) and updates a progress indicator.

### Option B: Generate script

Writes a self-contained `.sh` or `.py` script to a user-chosen path:

```bash
#!/bin/bash
# Generated by ultrack_wrapper
# Run this on your cluster or local machine
set -e
# conda activate myenv  # uncomment and edit if needed

INPUT_DIR="/path/to/raw/"
OUTPUT_DIR="/path/to/cellpose_output/"

for t in 0 1 2 3 4; do
  python -m ultrack_wrapper.stages.s01a_cellpose_nucleus \
    --input-dir "$INPUT_DIR" \
    --output-dir "$OUTPUT_DIR" \
    --model nuclei --diameter 17 --anisotropy 1.0 --min-size 500 \
    --timepoints $t
done
```

The user copies this to their cluster and runs it. When done, they point the
plugin at the output directory and continue with s02.

## Widget layout

Single docked panel with tabs:

### Tab 1: Project

- **Project directory** — root folder for all inputs/outputs
- Position selector
- Timepoint range (auto-detected or manual)
- Load / Save config

### Tab 2: Cellpose (s01a + s01b)

- Model, diameter, anisotropy, min_size parameters
- Input directory (auto-resolved from project, or manual override)
- Output directory
- **Run mode**: "Open Terminal" / "Generate Script"
- Run button + progress (file count polling)
- "Load Results" button → adds prob/flow layers to viewer

### Tab 3: Foreground (s02)

- Input directory (cellpose output)
- Threshold method: Fixed / Otsu / Triangle
- Threshold value slider (for fixed mode)
- **Preview** button → shows foreground overlay for current timepoint
- **Run All** button → processes all timepoints, writes `foreground.tif`
- Results automatically added as Labels layer

### Tab 4: Tracking (s03)

- Ultrack parameters (min_area, max_area, min_frontier, max_distance, etc.)
- Run button (threaded, with progress)
- Results loaded as Labels + Tracks layers

### Tab 5: Post-processing (s04 + s05)

- 2D projection parameters
- Cell expansion parameters
- Run buttons
- Results loaded as Labels layers

## Implementation order

1. **Project scaffold** — `pyproject.toml`, package structure, `napari.yaml`
2. **s02_foreground.py** — core logic (flow magnitude + thresholding)
3. **Foreground widget** — napari widget for s02 with preview + run
4. **s01a standalone script** — CLI entry point for Cellpose 3D
5. **Terminal launcher + script generator** — runners for s01a
6. **Cellpose widget tab** — parameter UI + run mode selector
7. **s03_tracking.py** — Ultrack integration
8. **Tracking widget tab**
9. **s04 + s05** — projection and cell expansion
10. **Post-processing widget tab**
11. **Project tab** — config management, directory resolution

---

## Key differences from cellpose_segmentation

- **Self-contained**: no dependency on or import from `cellpose_segmentation`
- **Simpler s02**: foreground detection via flow magnitude thresholding instead
  of multi-hypothesis `labels_to_contours`. Much faster and more interpretable.
- **No `pipeline.json` lineage system** (for now): simpler directory-based I/O.
  Each stage reads from a user-specified input dir and writes to an output dir.
- **Interactive parameter tuning**: the user can preview results in napari before
  committing to a full run
- **s01a is always external**: never runs in the napari process
